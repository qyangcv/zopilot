# v0.3.0 核心方向

v0.3.0 要实现两个核心能力：

1. 更好的文件内容检索：用稳定的文档处理与证据检索流程替代当前 `paper_read` 的临时全文切分和词法打分。
2. 工作区概念：支持 item、collection、library 三层使用场景，让会话、可选资料范围和 UI 入口有清晰归属。

Zopilot 的定位从 Zotero 侧栏 / Codex wrapper 升级为 **论文处理引擎 + agent runtime**。Zotero 提供原始资料，Zopilot 负责资料管理、解析、索引、检索、证据组织、workspace 和会话；Codex 只是当前 agent，未来应能替换或扩展到 BYOK 模型。

宏观路线是：Zopilot 自己负责把原始资料处理成可检索、可引用的证据，agent 只通过工具读取这些证据，而不是直接拿到 PDF 路径后自行解析。这样可以把解析、索引、证据格式和权限边界保留在 Zopilot 内部，更适合长期支持多模型、多资料、多 workspace。

数据模型从 **Paper-first** 转向 **Source-first**。Source 是底层可处理资料单元，可以来自 Zotero PDF、附件、上传文件、选中片段或未来其他来源。Conversation 不再直接等同于“某篇论文的聊天”，而是引用 workspace 与本轮选择的 sources。

核心对象边界：

- Workspace：会话与任务范围边界，决定会话归属、默认上下文、可 `@` 选择的 source universe 和 UI 入口。
- Source：跨 workspace 共享的原始资料资产，是解析和索引的输入。
- Material：Source 被处理后的稳定内部文档表示。
- Retrival：围绕 Material 建立的检索资产，跨 workspace 共享。
- Context：根据 workspace、@ mentions 和用户问题组织出来的上下文包，供 agent 使用。
- Agent Runtime Adapter：连接 Codex 和未来 BYOK provider。

Workspace 采用三层模型：

- Item Workspace：单篇 Zotero item / paper 的工作区，会话默认围绕当前 item。
- Collection Workspace：Zotero collection 的工作区，可在该 collection 及子 collection 范围内选择一篇或多篇资料。
- Library Workspace：整个 Zotero library 的工作区，用户手动选择需要进入上下文的资料。

关键原则：

- Workspace 不是解析和索引的存储边界。
- Source、Material、Retrival 应跨 workspace 复用。
- Collection workspace 默认递归包含子 collection。
- 插件内优先通过 Zotero API 读取 library、collection、item 和 attachment。
- MCP / tool surface 只暴露 agent 需要表达的阅读和检索意图，具体解析、索引、排序和证据组织留在 Zopilot 内部。

模块解耦要求：

- Tool 只负责 agent 可见接口，不实现解析、检索和排序。
- Workspace 只负责会话归属和 source 范围，不拥有 Material 或 Retrival。
- Source 只表示原始资料，不包含解析结果。
- Material 只表示处理后的文档，不绑定具体检索算法。
- Retrival 只负责可替换的检索能力，对外提供稳定查询接口。
- Context 只负责组装给 agent 的上下文，不关心底层检索算法。

建议的实现顺序：

1. 先明确 `Workspace`、`Source`、`Conversation` 的数据模型和存储布局。
2. 再把现有 `paper_read` 收敛成薄 tool facade，内部改走 Context Builder。
3. 然后补齐 Material、Retrival 和 Source-first 检索流程。
4. 最后扩展 workspace UI、`@` 选择、多 source 检索和 BYOK agent adapter。
