# v0.3.0 Light 宏观方案

本文档是 v0.3.0 的轻量宏观方案，与 `notes/light-solution-for-doc.md` 对齐。目标是把 Zopilot 从 Zotero 侧栏里的 Codex wrapper 推进到一个轻量的 **Source -> Material -> Retrieval -> Context -> Agent** 论文处理管道。

## 核心目标

v0.3.0 light 只做两件事：

1. **稳定的文件内容检索**：替代当前 `paper_read` 的 Zotero full text + 临时切分 + 简单词法打分。
2. **清晰的 workspace 边界**：支持 item、collection、library 三层使用场景，让会话归属、source 范围和 UI 入口明确。

本版本不做 heavy local RAG：不引入 OCR、embedding、语义检索、向量数据库、本地 reranker、SQLite FTS5、Docling 或长期运行的 sidecar service。

## 定位

Zotero 提供原始资料和元数据。Zopilot 负责资料管理、解析、缓存、chunking、artifact registry、FlexSearch 检索、证据组织、workspace 和会话。Codex 是当前 agent，未来可通过 Agent Runtime Adapter 替换为 BYOK provider。

agent 不直接解析 PDF，也不直接访问底层索引。agent 只通过 tool 表达阅读或检索意图，由 Zopilot 返回可追踪的 Context。

## 核心对象

- **Workspace**：会话和任务范围边界，决定默认上下文、可选 source universe、`@` 选择范围和 UI 入口。
- **Source**：原始资料资产，是解析和索引输入。首个落地 source 是 Zotero PDF attachment，后续可扩展到其他附件、上传文件或选中片段。
- **Material**：Source 被处理后的稳定内部表示，包括 Markdown、page chunks、assets、manifest、chunks 和 artifacts。
- **Retrieval**：围绕 Material 建立的检索能力。v0.3.0 light 固定实现为 FlexSearch，但对外保持稳定查询接口。
- **Context**：根据 workspace、用户问题和选定 sources 组装出的证据包，供 agent 使用。
- **Agent Runtime Adapter**：连接 Codex 和未来 BYOK provider。

## Workspace 模型

- **Item Workspace**：单篇 Zotero item / paper 的工作区，会话默认围绕当前 item。
- **Collection Workspace**：Zotero collection 的工作区，可在该 collection 及子 collection 范围内选择一篇或多篇资料。
- **Library Workspace**：整个 Zotero library 的工作区，用户手动选择需要进入上下文的资料。

Workspace 不是解析和索引的存储边界。Source、Material 和 Retrieval cache 应跨 workspace 复用。

## Light 文档管道

v0.3.0 light 的论文处理链路固定为：

```text
Zotero PDF attachment
  -> Source resolver
  -> PyMuPDF4LLM / PyMuPDF
  -> Material cache
  -> section-aware chunks
  -> artifact registry
  -> FlexSearch retrieval
  -> RRF + structural boost + MMR
  -> Context packing
  -> Agent answer
```

技术选型：

- PDF 解析：`PyMuPDF4LLM`
- PDF 页面渲染和局部截图：`PyMuPDF`
- 本地存储：Zopilot cache 目录中的 Markdown、JSONL、assets、manifest
- Chunking：Markdown section-aware chunker
- Artifact registry：自定义图、表、公式、页面资产索引
- Retrieval：`FlexSearch`
- Fusion / diversity：RRF + structural boost + MMR

FlexSearch index 可以按需在内存中重建。跨 workspace 复用的稳定资产是 Material cache、chunks、artifacts 和 manifest，不是某个活跃内存 index 实例。

## Tool 边界

Tool 是 agent 可见接口，只负责表达意图和返回 Context，不实现解析、检索、排序或缓存策略。

推荐方向：

- 现有 `paper_read` 收敛成薄 facade。
- 内部统一调用 Context Builder。
- Context Builder 根据 workspace、selected sources、query router 和 retrieval result 组织证据。
- tool result 返回证据、locator、页码、artifact path、warnings，而不是最终回答。

## Context 要求

Context 必须可追踪、可压缩、可解释：

- 每条证据保留 source identity、page、section path、chunk/artifact ID。
- 图表类问题返回 caption、附近正文、图片路径和解析 warning。
- 普通文本问题返回结构化 snippets，而不是整篇 Markdown。
- Context packer 控制最大字符数、chunk 数、artifact 数和同页重复数量。

Codex/Agent 负责最终理解、图片分析和答案生成。Zopilot 负责让 agent 拿到正确材料。

## 实现顺序

1. 明确 `Workspace`、`Source`、`Conversation` 的数据模型和存储布局。
2. 将 `paper_read` 改为薄 tool facade，内部接入 Context Builder。
3. 实现 Zotero PDF attachment 的 Source resolver。
4. 实现 `PyMuPDF4LLM + PyMuPDF` 的 Material 构建和 cache invalidation。
5. 实现 chunks、artifact registry 和 FlexSearch retrieval。
6. 实现 query router、RRF、structural boost、MMR 和 Context packing。
7. 扩展 workspace UI、`@` source 选择和多 source 检索。

## 成功标准

v0.3.0 light 成功时，Zopilot 应能：

- 不依赖 Zotero full text 作为主 source。
- 对当前 paper 构建可复用 Material。
- 回答普通文本类问题时返回稳定证据 snippets。
- 回答 `Table/Figure/Formula/Page` 类问题时返回 locator、caption、附近文本和图片路径。
- 让 item、collection、library 三类 workspace 使用同一套 Source/Material/Retrieval/Context 边界。
