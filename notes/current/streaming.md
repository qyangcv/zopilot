# Better Chat Message Streaming

## 目标

- 更低的资源占用：减少流式输出期间不必要的计算和界面更新，显著降低 CPU 消耗。
- 更流畅的体验：输出连续、稳定，并让显示速度与模型实际生成速度保持平衡。
- 更广的兼容性：支持 Windows 和 macOS，兼容不同模型与服务商的流式输出方式。
- 更高的代码质量：用低复杂度的统一架构实现流式输出，减少高耦合和重复逻辑，提高可维护性与可扩展性。

## 技术选型

采用参考 VS Code Chat 的分层流式架构，并针对 Zopilot 的运行环境进行轻量化实现：

- 使用统一的流式事件模型承接不同后端和模型的输出。
- 使用独立的运行中消息状态，将高频更新与普通界面状态隔离。
- 使用统一刷新调度器合并高频事件，平衡响应速度与资源消耗。
- 将消息拆分为具有稳定身份的内容块，只更新实际发生变化的部分。
- 使用独立的滚动状态管理，协调自动跟随与用户主动浏览。
- 优先使用跨平台标准能力，不引入平台专用实现。
- 建立纯代码驱动的性能测试工具，自动完成固定场景回放、资源采样和结果判定。

## 宏观方案

```text
不同模型与服务商的流式输出
            ↓
统一事件归一化
            ↓
权威运行状态
            ↓
统一刷新调度
            ↓
消息与内容块局部更新
            ↓
滚动与视觉状态协调
```

流式事件首先被转换为统一格式，再写入唯一的权威运行状态。界面更新由单一调度器负责，不直接跟随每个上游事件刷新。历史消息和普通界面保持稳定，只有当前发生变化的消息内容块参与更新。

流式开始、进行、完成、中断和异常均使用同一套生命周期与数据流。不同模型的输出差异只在输入适配层处理，不向界面层扩散。遇到乱序、非连续更新或不支持的输出形式时，统一回退到安全的完整状态更新。

## 质量原则

- 单一事实来源：运行状态只有一个权威来源。
- 单一更新通道：所有流式界面更新经过同一个调度器。
- 局部变化原则：没有变化的消息和内容块不参与更新。
- 职责分离：输入兼容、状态管理、刷新节奏、界面更新和滚动控制相互独立。
- 自动化验证：测试与验收可通过标准命令独立运行、稳定复现并生成可比较的结果。

## 验收方向

- 流式输出期间的平均 CPU、峰值 CPU 和完成后的空闲 CPU 明显优于当前实现。
- 快速、缓慢、突发和间歇式输出均保持稳定，不出现明显卡顿、跳变或失控追赶。
- 历史消息和无关界面不随流式增量重复更新。
- 切换会话、停止生成、异常中断和后台恢复后，消息顺序与最终内容保持正确。
- 核心逻辑保持单向数据流，不同时维护多套高耦合的流式更新机制。
- 性能测试使用固定场景和统一指标，通过纯代码自动运行并输出机器可读报告。
- 性能验收不依赖人工操作或 Computer Use 控制 Zotero；界面人工检查只作为非性能项的补充。
